## 書き分け原則

- **恒久的に参照される仕様**は `docs/features/{feature-name}.md` に書く（実装完了後も正本として残る）
- **今回の差分にしか意味のない情報**（変更対象ファイル・受け入れ条件・「スキーマ変更なし」等の差分注記）は `docs/tasks/` の実装指示書に書く（実装完了・検証後に削除する一時文書）
- feature 文書は現在形の事実として書く。「新規〜なし」「〜変更なし」のような差分視点の記述を残さない

## 手順

### 1. docs/features/ に仕様を書く

新機能なら `docs/features/{feature-name}.md` を作成する。既存機能の変更なら該当ファイルを更新する。

仕様ドキュメントには以下を含める:
- **概要** — 何をするか 1〜2 文
- **認可** — 誰が実行できるか（原則全メンバー開放。例外は理由とともに明記。docs/design.md「認可ポリシー」）
- **実行コンテキスト** — guild 限定 / DM 可（`docs/core.md` の実行コンテキスト表を更新する）
- **コマンド** — スラッシュコマンドの名前・引数・挙動・エラーケース（応答メッセージは完全な文字列で）
- **公開範囲** — 各応答が ephemeral か public か
- **副作用の順序と補償** — Discord リソース操作・DB 更新の実行順序、失敗時の補償・部分成功の扱い
- **冪等性・送達保証** — 定期ループから呼ばれる処理のリトライ挙動と通知の重複防止（docs/design.md「定期ループから呼ばれる処理は冪等にする」に整合させる）
- **同時実行** — 競合時の挙動（DB 制約・ロック・二段チェック）
- **上限** — 入力の文字数・件数、出力の分割・切り詰め規則
- **データモデル** — dataclass 定義（必要な場合。`docs/data-contracts.md` に追記して feature 文書から参照する）
- **DB スキーマ** — テーブル定義・migration（必要な場合。同上）
- **Embed / メッセージ形式** — Discord に送信するメッセージの形式
- **関連設定** — 環境変数（必要な場合。docs/design.md「設定値の検証契約」に従う）
- **対象外** — 機能としての恒久的な非目標（今回やらないだけのものは docs/tasks/ の「スコープ外」へ）

該当しない項目は省略してよいが、認可・実行コンテキスト・公開範囲・副作用の順序・上限は原則省略しない。`docs/features/ctf-team.md` を詳細な参考例、`docs/features/times.md` を最小の参考例として使う。

### 2. AGENTS.md の参照テーブルを更新する

`## 仕様リファレンス` のテーブルに新しいドキュメントへのポインタを追加する。

### 3. docs/ の鮮度管理（文書が正本）

- 実装の有無に関わらず、仕様の正本は docs/ である。文書だけで実装を完了できる自己完結な内容にする
- データモデル・DDL・Database メソッド・設定（環境変数）の契約は `docs/data-contracts.md` に集約し、feature 文書からは参照する
- **実装完了後も文書を正本とする。** 文書と実装が乖離した場合は原則として実装を文書へ合わせる。仕様変更が妥当な場合は、先に文書を更新してから実装を追随させる
- 設計変更を行った場合、対応する docs/ を同じ変更内で更新する

### 4. docs/tasks/ に実装指示書を書く

`docs/tasks/YYYY-MM-DD-{name}.md`（段階化する場合は `YYYY-MM-DD-phaseN-{name}.md`。CLAUDE.md「リファクタリングの段階化」）を作成する。その文書だけで実装を完了できる自己完結な内容にし、以下の構成で書く:

- **背景（何が問題か / 何を追加するか）** — feature 文書のどの仕様に対応する差分か
- **対象ファイル** — 変更するファイルの列挙（触れないものも必要なら明記）
- **実装内容** — 具体的な変更手順・コード（仕様は feature 文書を参照し、重複記述を最小にする）
- **受け入れ条件** — 実装完了を判定できるテスト観点（docs/design.md「境界変更のテスト方針」参照）と検証コマンド（`AGENTS.md`「検証」）
- **スコープ外** — 今回やらないこと（後続 phase の内容を先取りしない）
- **周辺更新の要否** — README・`.env.example`・DB migration・`tests/test_architecture.py` の feature 一覧

実装指示書は一時文書であり、実装完了・検証後に削除する。

### 5. Codex 向けのプロンプトを出力する

文書を書いた後、ユーザーが Codex に渡せるプロンプトを会話内に出力する。形式:

```
docs/tasks/YYYY-MM-DD-{name}.md の指示に従って実装してください。
仕様の正本は docs/features/{feature-name}.md。
AGENTS.md の制約・規約・検証手順に従うこと。
```
