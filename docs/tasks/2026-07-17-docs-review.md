## 2. 矛盾・プラットフォーム制約違反（S1–S11）

文書の記述どおりに実装すると誤る、または記述同士が両立しない箇所。**全件が文書修正のみで解消できる。** 外部仕様に依る判定（S1・S2・S3・S5）には公式資料リンクを付した（後続 AI の再検証コスト削減のため）。

### S1. voice チャンネル名が Discord の 100 文字上限を超え得る 【B・公式仕様確認済み】高
`ctf-team.md:174-178`。normalize が 100 文字に切った**後**、voice は `{normalize結果}-voice` を base とする（最大 106 文字）。`pick_unique_channel_name` の切り詰めは一意化 suffix（`-2` 等）付与時のみで、`-voice` には適用されない。Discord のチャンネル名上限は 100 文字。「`-voice` 込みで 100 文字に収める」規則を明記する。
（Discord Channel Resource: https://docs.discord.com/developers/resources/channel）
→ **対応済み（手順 3）**: 実装確認の結果、`pick_unique_channel_name` は**全候補を 100 文字に切り詰めており実装は上限を超えない**（切り詰めが suffix 時のみという文書の記述が不正確だった）。文書修正のみで解消し、実装追随は不要。

### S2. AlpacaHack 週次 Embed が Embed 上限（合計 6000 文字・field name 256 文字）を超え得る 【B+V・公式仕様確認済み】高
`alpacahack.md:89-101`。最大 24 field × 各 value 1024 文字（＋name）は、field 単体の制限は守るが **Embed 全体の 6000 文字制限**を大幅に超え得る。さらに username に長さ制限が無いため（al-4）、field name `{username} ({n} solves)` が **field name の 256 文字上限**も超え得る（V 指摘）。合計・field name の双方への切り詰め規則を定義する。
（Discord Message Resource: https://docs.discord.com/developers/resources/message）

### S3. `INSERT OR IGNORE` と「False = entry_id 重複」契約の不一致 【B・公式仕様確認済み】高
`audit-log.md:41,45`。SQLite の `OR IGNORE` は対象の UNIQUE 以外にも、NOT NULL・CHECK・他の UNIQUE 等の**適用可能な制約違反を無視し得る**（外部キー違反には適用されない）ため、「False は entry_id 重複スキップのみ」という戻り値契約を保証しない。`ON CONFLICT(entry_id) DO NOTHING` のように conflict target を限定する仕様へ改める。
（SQLite ON CONFLICT: https://www.sqlite.org/lang_conflict.html）

### S4. `/sudo` 手順 4 と手順 5 の付与対象決定が矛盾 【B・検証済】高
`sudo.md:32-33`。手順 4 が**現在の** `ADMIN_ROLE_ID` の存在を先に要求し、手順 5 が「付与対象は grant レコードの保存済み `role_id` を維持」と定める。設定変更後に現行設定のロールが削除済みで、保存済みロールが健在の場合、手順 5 の方針では成立するのに手順 4 が「付与対象のロールが見つかりません。」で中断する（エラーメッセージの指すロールも実際の付与対象と異なる）。**先に grant を読んで対象 ID を決定し、その対象を解決する**順序に改める。

### S5. `COLLATE NOCASE` は ASCII 26 文字のみ 【B・公式仕様確認済み】中
`ctf-team.md:234` は「大文字小文字無視」と無限定に説明するが、SQLite 組み込み NOCASE の case folding は ASCII のみ。Unicode の CTF 名も対象にするなら正規化済み canonical key が必要。少なくとも「ASCII のみ」と明記する。
（SQLite Datatypes / Collating Sequences: https://www.sqlite.org/datatype3.html）

### S6. sudo エラーメッセージのバッククォート有無が非一意 ◎ 高
`sudo.md:37,49` は「…確認してください(`` `/perms` ``)。」、`sudo.md:103-104` の表は `(/perms)`。Discord 上で見た目が変わるため「完全な文字列」（SKILL.md:14 の要求）として確定していない。表と手順の重複自体も解消する（どちらかを正本に）。

### S7. `CTF_TEAM_CATEGORY_ID` の説明と募集メッセージの投稿先が矛盾 【B・検証済】中
`ctf-team.md:242` は「**募集メッセージ**・discussion・voice を作成するカテゴリ」とするが、募集メッセージは `#role` チャンネルへ投稿される（`ctf-team.md:26,170,254`）。説明の修正、または `#role` をそのカテゴリ内から探す仕様ならその明記（ct-4 の検索範囲問題と同時に解決）。

### S8. 「文字列系は空文字→デフォルト」は必須文字列に適用できない 【B・検証済】中
`design.md:190`。`DISCORD_TOKEN` にはデフォルトが無く、空はフォールバックでなく起動拒否のはず。「必須文字列 / 既定値付き文字列 / optional 文字列」の 3 分類に書き分ける（P3 で実施）。

### S9. 例外階層の catch 順・メッセージ言語規約の欠落 ◎（言語不統一は A）高
`AGENTS.md:50` で `ExternalAPIError` は `ServiceError` の派生。`design.md:240-246` は両者に別処理（そのまま表示 vs フォールバック応答）を割り当てるが、**先に個別 catch する規則がどこにも無く**、`AGENTS.md:39` どおり `except ServiceError` で統一すると `ctftime.md:43-44` の英語メッセージ（`"Unexpected CTFtime response."`）がユーザーへ露出する。さらに `alpacahack.md:47` の同種例外は日本語で、**例外メッセージの言語規約が feature 間で不統一**。例外表に「catch 順」「ExternalAPIError のメッセージは内部用（表示しない）」を明記する。

### S10. blocking I/O 規則の適用範囲が 3 文書で不一致 ◎ 中
`AGENTS.md:41` は「イベントループ外（起動時初期化・同期テスト）は対象外」と正確だが、`design.md:72`「blocking I/O は**全て** to_thread 経由」と `CLAUDE.md:69` は適用除外が脱落した過大な要約になっている。
